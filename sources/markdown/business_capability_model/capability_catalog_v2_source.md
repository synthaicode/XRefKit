# 能力定義カタログ v2

## 概要

本カタログは、各作業に必要な能力を定義し一覧化する。

**設計原則**：AI は作業領域を絞るほど性能が上がる。  
能力定義は AI エージェントのスコープを限定し、精度を最大化するための設計単位となる。  
能力は全体のリソースプールとして管理し、グループ・作業への割り当てに利用する。

---

## A. 能力と責務の境界定義

| 概念 | 定義 | 特徴 |
|------|------|------|
| **能力** | 入出力・制約・前提条件を持つ**再利用可能な作業単位** | 複数のグループ・フェーズで使い回せる。AI エージェントに割り当てる最小単位 |
| **責務** | 同じ能力を、ある業務目的に対して**どのように発揮するかを定めた運用上の定義** | 能力自体は共通でも、責務が変われば発揮のされ方・期待される出力・判断基準が変わる |

> 専門家は能力を保有しており、その能力を責務に応じた形で発揮する。  
> たとえば会計専門家は同じ会計能力を持っていても、財務を担うのか、経理を担うのか、管理会計を担うのかで、能力の発揮のされ方が変わる。  
> グループはミッションに応じて責務を定義し、責務に応じて必要な能力の組み合わせと発揮のされ方を定める。

### A-1. 能力のチューニング

能力はそのまま使うだけでなく、技術・業界・品質観点によって**チューニング**される。

- **ベース能力**：再利用可能な専門能力そのもの
- **チューニング**：能力を特定の技術・業界・観点向けに最適化すること
- **責務**：チューニングされた能力を、業務目的に対してどう発揮するか

**例1：会計**

- ベース能力：会計
- チューニング：建設業界会計
- 責務：財務、経理、管理会計

**例2：開発**

- ベース能力：ソフトウェア開発
- チューニング：C#
- 責務：実装、単体テスト、レビュー

> 同じ能力であっても、チューニングが変われば参照すべき知識・判断基準・期待される出力が変わる。  
> 同じチューニングであっても、責務が変われば能力の発揮のされ方が変わる。

## B. 判定能力の根拠必須ルール

**判定能力（作業種別：判定）の全てに適用する。**

評価結果には必ず根拠を付属させる。根拠のない評価は「わからない」として扱う。

| 必須項目 | 内容 |
|---------|------|
| **根拠参照** | 判定の拠り所となった上位成果物・ドメイン知識の名称と参照箇所 |
| **項目別根拠** | 複数の判定項目がある場合、項目ごとに根拠を紐付ける |
| **根拠不明時** | 根拠を特定できない項目は「わからない」として記録し判定しない |

---

## C. 管理系能力の更新対象ルール

**管理能力（作業種別：管理）の全てに更新先を明示する。**

| 更新先 | 条件 |
|--------|------|
| **作業管理リポジトリ** | 案件ごとに変化する状態・進捗・判定・反省の記録 |
| **ドメイン知識リポジトリ（昇格候補）** | 再利用可能と判断された知見が昇格条件を満たした場合（→ プロトコル 4-5 参照） |

---

## D. 前提条件・発火条件について

**前提条件**：この能力を発火させる前に満たされていなければならない状態。  
**発火条件**：この能力が呼び出されるトリガーとなるイベントまたは状態。

親エージェント（オーケストレーター）はこれらを参照して能力の呼び出し順序を自動構成する。

## E. 能力 ID 命名規則

```
CAP - {カテゴリ} - {連番（3桁）}
```

| カテゴリ | 略称 | 対象作業 |
|---------|------|---------|
| 調査 | INV | 既存システムの調査・解析 |
| 供給者・予算 | SUP | 外部サービス・コスト関連 |
| 見積・前提 | EST | 見積・前提整理 |
| 要件定義 | REQ | 要件の定義・草案作成 |
| 計画 | PLN | タスク分解・計画立案 |
| 製造 | MFG | 実装・テスト・計測 |
| 品質管理 | QA | レビュー・判定・チェック |
| 事業評価 | BIZ | 価値・制約・優先順位の評価（CAB） |
| 運用 | OPS | リリース計画・監視設計 |
| 管理（共通） | MGT | 進捗・記録・差し戻し・エスカレーション |

> 新規追加時は既存の連番の末尾に追加する。廃止能力の番号は欠番とし、再利用しない。

## 能力の構造（フィールド定義）

```
能力
├── 能力ID           一意の識別子（命名規則に従う）
├── 能力名           短い名称
├── 作業種別         実施 / 判定 / 管理 のいずれか
├── 前提条件         発火前に満たすべき状態（D 参照）
├── 発火条件         呼び出されるトリガー（D 参照）
├── 入力             この能力が受け取るもの
├── 出力             この能力が生成するもの
│   └── 判定能力は根拠参照を必須とする（B 参照）
├── 必要ドメイン知識  ドメイン知識リポジトリから投入すべき情報
├── 更新先           管理能力のみ明示（C 参照）
├── 制約             作業範囲・禁止事項
└── 割り当て先       使用するグループ・作業フェーズ
```

**作業種別の定義**

| 種別 | 定義 | 追加ルール |
|------|------|-----------|
| 実施 | 成果物を生成する作業（実装・分析・草案作成等） | — |
| 判定 | 成果物の合否・適合性を評価する作業 | 根拠参照必須（B 参照） |
| 管理 | 手順・進捗・一覧を制御する作業 | 更新先を明示（C 参照） |

## PMBOK 5プロセスのタスク内適用パターン

PMBOK の5プロセスはプロジェクト全体だけでなく、**個々のタスク自体にも適用される**。  
各タスクはこのパターンを内包する。規模に応じて各プロセスの厚みが変わるが、省略はしない。

### 汎用タスクパターン（全タスク共通）

| PMBOK プロセス | タスク内で行うこと | 対応する仕組み・能力 |
|-------------|-----------------|-------------------|
| **立ち上げ** | タスクの目的・入力・完了条件を確認する。上位成果物（前工程の出力）の存在を確認する | 能力の前提条件チェック。不足があれば「わからない」として記録し立ち上げを保留 |
| **計画** | 作業一覧・対象一覧を作成する。管理テーブルの初期値を生成する | CAP-MGT-001（実行管理）：管理テーブル（作業×対象）の生成 |
| **実行** | 各作業リストを順に実施する。ステータスをチェックし不確実を明示する | 各 CAP-* 能力を発火　CAP-MGT-002：わからない／対象外を記録 |
| **監視・コントロール** | 管理テーブルを確認する。漏れを検出し担当へ差し戻す。確信度・コンテキストを記録する | CAP-QA-004（管理テーブルチェック・差し戻し）　CAP-MGT-002（メトリックログ記録） |
| **終結** | 全行が完了／わからない／対象外のいずれかになったことを確認する。未解決・対象外を次工程へ引き継ぐ。エスカレーション確認（スコープ変更・受容判断の要否） | CAP-QA-004（完了確認）　CAP-MGT-002（未解決リスト・引き継ぎ記録）　**CAP-MGT-003（対象外エスカレーション）** |

### 具体例：① 要求から見積を作成するタスク

| PMBOK プロセス | このタスクで行うこと | 成果物・引き継ぎ |
|-------------|-----------------|----------------|
| **立ち上げ** | 要求・変更対象一覧・供給者定義・予算定義の存在を確認する。不足があれば「わからない」として記録し保留する | — （前提条件未充足なら開始しない） |
| **計画** | 作業一覧（供給者四条件照合・コスト試算・解決案生成）と対象一覧（要求項目ごと）を作成し管理テーブルを初期化する | 管理テーブル（初期値） |
| **実行** | 各作業・各対象に対して CAP-SUP-001・CAP-SUP-002・CAP-EST-001 を発火し結果を記録する。不確実はその都度「わからない」として記録する | 問題点リスト・コスト試算書・解決案（複数）・前提リスト |
| **監視・コントロール** | 管理テーブルを確認し未実施を検出して差し戻す。確信度2以下の項目を「わからない」に格上げする | 差し戻し記録・メトリックログ |
| **終結** | 全行のステータス確定を確認する。完了分は見積書として次工程へ、わからない分は前提リストとして②へ引き継ぐ。対象外は CAP-MGT-003 でコーディネーターへ報告する | 見積書・前提リスト（未確定）・対象外リスト |

### 適用原則

- **省略禁止**：全タスクで5プロセスを適用する。特に「終結」の完了確認とエスカレーション確認を省略すると漏れが次工程へ流れる
- **監視・コントロールは実行と並行**：実行後にまとめて確認するのではなく、各作業の完了のたびにチェックする
- **立ち上げの保留**：前提条件を満たさない状態で能力を発火させない。これがタスクレベルの立ち上げ管理である
- **終結の二段確認**：完了確認（全行がステータス確定）とエスカレーション確認（スコープ変更・受容判断の要否）の両方を行う
